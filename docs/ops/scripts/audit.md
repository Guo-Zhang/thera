# Scripts 模块

## 设计思路

### 1. 定位转变

将 `activity/thera.py` 重构为独立的审计脚本，反映了对模块职责的重新理解：

- **Activity vs Script**: Activity 是运行时动态切换的执行流（与 Domain 概念重复），而审计脚本是一次性执行的独立工具
- **运行时 vs 离线**: Activity 参与应用的交互式运行，脚本则是离线分析工具

### 2. 目录重构

```
src/thera/activity/thera.py → scripts/audit.py
data/activity/thera/        → docs/ops/reports/
```

**原因**：
- `activity/` 目录已移除 ActivityManager，保留 `memo.py` 作为数据处理 pipeline
- `scripts/` 是独立工具的标准存放位置
- `docs/ops/reports/` 更适合审计报告这类运维输出（运维文档目录更符合语义）

### 3. 命名规范

- `audit.py`: 审计类脚本的命名，描述功能而非实现细节
- 输出文件使用描述性命名（如 `report.json`）

### 4. 模块导入

由于脚本位于包外，需要手动添加 `src` 到 `sys.path`：

```python
import sys
from pathlib import Path
sys.path.insert(0, str(Path(__file__).parent.parent / "src"))
```

## Audit 脚本设计意图

### 核心理念：文档的"单元测试"

传统测试验证代码正确性，Audit 验证**决策质量**。AGENTS.md 是项目知识的集中体现，审计它就是在评估项目"思维框架"的成熟度。

### 元认知浓度分析

元认知 = "对思考的思考"。高元认知浓度的文档具备：
- 自我意识：明确描述自身职责和边界
- 策略意识：包含方法论和最佳实践
- 监控意识：有进度追踪和质量标准
- 评估意识：定义了成功指标
- 调整意识：包含迭代优化机制

### 与传统测试的区别

| 维度 | 传统测试 | Audit |
|------|----------|-------|
| 对象 | 代码正确性 | 文档质量 |
| 标准 | 通过/失败 | 评分制 (1-10) |
| 目的 | 验证功能 | 评估思维 |

### 使用场景

1. **发布前检查**: 每次重要更新前审计 AGENTS.md，确保文档质量
2. **趋势追踪**: 保存历史报告，对比元认知评分变化
3. **改进依据**: 依据 `recommendations` 迭代优化文档

### 可扩展方向

- 批量审计多个文档（AGENTS.md + docs/dev/*）
- 对比分析（多个模块的元认知水平对比）
- 自定义评分维度

## 运维文档规范

### 定位

`docs/ops/` 是运维类文档的存放目录，包括：
- 审计报告（`reports/`）
- 运维脚本说明（`scripts/`）
- 系统健康检查报告
- 部署和运维手册

### 使用规范

1. **报告命名**: 使用 `report.json` 作为默认输出，由审计脚本自动生成
2. **历史记录**: 每次审计覆盖上一次报告，如需保留历史可手动复制
3. **更新时机**: 
   - 重大功能变更后运行审计
   - 定期（如每周/每月）检查元认知趋势
   - 发布前质量门槛检查

### 维护规范

1. **报告归属**: 审计报告属于项目运维资产，纳入版本控制
2. **文档同步**: 当审计评分发生显著变化（±2分）时，需同步更新 AGENTS.md
3. **趋势分析**: 可手动创建 `docs/ops/reports/history/` 目录保存历史报告进行对比
